# メルマガ配信計画 検証レポート

> 作成日: 2026-03-13
> 目的: 週3回（火・木・土）の定期メルマガが1年以上コンテンツを維持できるか検証

---

## 1. 提案する配信スケジュール

| 曜日 | コンテンツ | 自動生成元 |
|------|----------|-----------|
| **火曜** | 売れた速報（前日〜当日の売約＋残在庫数） | 依頼管理シート + データ1シート |
| **木曜** | ブランドコラム or マーケット動向 | 在庫データ + OpenAI API |
| **土曜** | 週末おすすめピックアップ（価格帯・カテゴリ切り口） | データ1シート全在庫 |

**目的:** 自社ブランド（デタウリ.Detauri）を思い出してもらうこと

## 3. 重複・衝突リスクの分析

### 3-1. 同日配信の衝突マトリクス

```
■ = 衝突あり  □ = 衝突なし  △ = 条件付き衝突

                新着通知  休眠ｸｰﾎﾟﾝ  月次ｻﾏﾘｰ  ﾌｫﾛｰｱｯﾌﾟ  入金ﾘﾏｲﾝﾄﾞ  ｶｺﾞ落ち
火:売れた速報    ■         △          □         △         △           △
木:ﾌﾞﾗﾝﾄﾞｺﾗﾑ    ■         △          ■(1日)    △         △           △
土:週末ﾋﾟｯｸｱｯﾌﾟ  ■         △          ■(1日)    △         △           △
```

### 3-2. 内容の類似性マトリクス

```
類似度: ◎=高  ○=中  △=低  ×=なし

              新着通知  月次ｻﾏﾘｰ  ﾌｫﾛｰｱｯﾌﾟ  休眠ｸｰﾎﾟﾝ  ｶｺﾞ落ち
火:売れた速報   ○        △        ×         ×          ×
木:ﾌﾞﾗﾝﾄﾞｺﾗﾑ   △        ◎        ×         ×          ×
土:週末ﾋﾟｯｸｱｯﾌﾟ ◎        ○        ×         ×          ×
```

**最大リスク:**
- **土曜「週末ピックアップ」と毎日「新着通知」** → どちらも「おすすめ商品の紹介」で内容が酷似
- **木曜「ブランドコラム」と月次「在庫サマリー」** → どちらもブランド×在庫の話題

### 3-3. 対策（実装必須）

| ルール | 説明 |
|--------|------|
| **1日1マーケメール制限** | 同日に新着通知を送信済みの顧客には新規メルマガをスキップ |
| **休眠クーポン優先** | 休眠クーポン送信日はメルマガスキップ |
| **フォローアップ優先** | フォローアップ対象者はメルマガスキップ |
| **月初スキップ** | 毎月1日は月次サマリーがあるためメルマガスキップ |
| **入金リマインド配慮** | 入金リマインド対象者はメルマガスキップ |

## 4. 1年間のコンテンツ持続性検証

### 4-1. 火曜「売れた速報」

**年間配信回数:** 約52回

**コンテンツ生成元:**
- 依頼管理シートの直近の売約データ（受付番号・商品名・ブランド・価格）
- データ1シートの残在庫数

**変動要素:**
- 売れた商品のブランド・カテゴリ・価格帯 → 毎回異なる
- 残在庫数の増減 → 仕入れサイクルに連動

**ネタ切れリスク: 低**

| シナリオ | 発生確率 | メール内容 |
|---------|---------|-----------|
| 売約あり（通常） | 高 | 「昨日○点が売約済み。△△ブランドが人気です。残り□□点」 |
| 売約なし | 中（閑散期） | **送信スキップ**（売約0件の速報は無意味） |
| 大量売約 | 低 | 「今週だけで○○点が売約！在庫が急減中」 |

**1年持続性: ○ 問題なし**
- 商品が売れる限りネタは自動発生する
- 売約なしの週は自動スキップで違和感なし
- 「何が売れているか」は毎週変わるため飽きにくい

**懸念:**
- 閑散期（1月後半、8月）に売約が少ないと配信頻度が落ちる
- → 対策: 売約なしの場合は「今週の注目在庫」に自動切り替え

---

### 4-2. 木曜「ブランドコラム / マーケット動向」

**年間配信回数:** 約52回（月初1日が木曜の場合は約48回）

**コンテンツ生成元:**
- データ1シートの在庫ブランド一覧
- ブランド一覧シートのカテゴリ分類（ハイブランド/デザイナーズ/セレクト）
- 注文データの売れ筋傾向
- GA4データ（流入・閲覧傾向）
- OpenAI API によるコラム自動生成

**ローテーション案（4パターン月替わり）:**

| 週 | パターン | 例 |
|----|---------|-----|
| 第1週 | ブランド深堀り | 「UNITED ARROWSの魅力と古着市場での立ち位置」 |
| 第2週 | カテゴリ動向 | 「ジャケットの引き合いが増加中。秋冬トレンド」 |
| 第3週 | 価格帯分析 | 「5,000円以下のお手頃在庫が充実。副業仕入れにおすすめ」 |
| 第4週 | 売れ筋ランキング | 「先月のTOP10ブランド。1位は○○」 |

**ネタ切れリスクの定量分析:**

| リソース | 規模 | 52回分のネタ |
|---------|------|-------------|
| 取扱ブランド数 | 動的（推定50〜100+） | ブランド深堀りだけで50回以上 |
| カテゴリ数 | 推定10〜15種 | カテゴリ × 季節で60パターン |
| 価格帯 | 5段階程度 | 価格帯 × カテゴリで50パターン |
| 売れ筋データ | 毎月更新 | 12ヶ月 × ランキング = 12パターン |

**1年持続性: ○ 問題なし**
- 4パターンローテ × ブランド/カテゴリの組み合わせで **理論上200パターン以上**
- AI生成（OpenAI）により同じブランドでも切り口を変えられる
- 季節変動で自然にネタが変わる（春夏→秋冬で需要品目が入れ替わる）

**懸念:**
- AI生成の品質が安定しない場合がある
- → 対策: テンプレートで構造を固定し、AIは「本文の読み物部分」のみ生成。データ部分は実数値を使用
- 同じブランドが短期間で繰り返し出ないよう、過去3ヶ月の配信履歴を参照して重複回避

---

### 4-3. 土曜「週末おすすめピックアップ」

**年間配信回数:** 約52回（月初1日が土曜の場合は約48回）

**コンテンツ生成元:**
- データ1シート全在庫から条件フィルタで抽出

**切り口ローテーション（6パターン）:**

| パターン | 抽出ロジック | 例 |
|---------|------------|-----|
| ブランド指定 | 在庫の多いブランドTOP1 | 「BEAMSの在庫が充実中 — 12点」 |
| カテゴリ指定 | 季節に合ったカテゴリ | 「アウター特集 — 寒さ対策に」 |
| 価格帯指定 | 特定価格帯 | 「3,000円以下のお手頃アイテム20選」 |
| サイズ指定 | 在庫多のサイズ | 「Mサイズ充実中 — 回転率の高いサイズ」 |
| 新着＋既存ミックス | 新着5 + 既存5 | 「今週の新着と見逃しアイテム」 |
| 状態指定 | 未使用・新品 | 「未使用タグ付きアイテム特集」 |

**ネタ切れリスクの定量分析:**

| 切り口 | バリエーション | 年間パターン数 |
|--------|-------------|-------------|
| ブランド | 50〜100+ | 50+ |
| カテゴリ | 10〜15 | 15 × 4季節 = 60 |
| 価格帯 | 5段階 | 5 × 12ヶ月 = 60 |
| サイズ | 5〜8種 | 40+ |
| 状態 | 3〜5種 | 15+ |
| **合計** | | **225パターン以上** |

**1年持続性: ○ 問題なし**
- 6パターンローテ × 各パターンの変数で年間225パターン以上
- 在庫が入れ替わるため、同じ切り口でも中身は毎回異なる

**懸念:**
- **新着通知との内容重複が最大リスク**（どちらも「商品を紹介する」メール）
- → 対策: 土曜ピックアップは「在庫全体から選定」「切り口を明示」で差別化。新着通知は「新しく入った商品」に限定。件名・デザインテンプレートも明確に分ける

## 5. 季節変動への対応

### 5-1. 古着卸売の季節サイクル

| 月 | 需要 | 在庫動向 | メルマガ切り口 |
|----|------|---------|-------------|
| 1月 | 低〜中（年始） | 冬物在庫豊富 | 新年挨拶＋冬物セール訴求 |
| 2月 | 中（春物準備） | 春物仕入れ開始 | 春物先取り＋冬物ラスト |
| 3月 | 高（新生活） | 春物増加 | 新生活需要＋春物特集 |
| 4月 | 高（新年度） | 春物ピーク | 新年度スタート＋軽アウター |
| 5月 | 中 | 春夏切替 | 初夏アイテム＋GW後の仕入れ |
| 6月 | 中（梅雨） | 夏物増加 | 梅雨対策＋夏物先取り |
| 7月 | 中〜高（夏） | 夏物ピーク | 夏物特集＋サマーセール |
| 8月 | 低（盆休み） | 秋物準備 | 秋冬先取り＋残暑セール |
| 9月 | 高（秋物） | 秋物増加 | 秋物特集＋衣替え需要 |
| 10月 | 高 | 秋冬ピーク | ハイブランドアウター＋冬準備 |
| 11月 | 高（冬物） | 冬物増加 | ブラックフライデー＋冬物特集 |
| 12月 | 中〜高（年末） | 年末在庫整理 | 年末セール＋年始準備 |

### 5-2. 閑散期（1月後半・8月）の対策

| 課題 | 対策 |
|------|------|
| 売約が少なく「売れた速報」のネタ不足 | 売約0件の週は「今週の注目在庫」に自動切替 |
| 在庫入替が少なくピックアップの鮮度低下 | 「過去に見逃したアイテム」「長期在庫のお買い得品」に切替 |
| 読者のメール開封率低下 | 配信頻度を週2に落とす（土曜スキップ） |

## 6. 「飽き」の検証：52週シミュレーション

### 6-1. 3ヶ月目（12週）の時点

| 曜日 | 配信済み回数 | 受信者が感じる印象 |
|------|-----------|------------------|
| 火:売れた速報 | 12回 | 「毎週違うものが売れてる、市場の動きがわかる」 |
| 木:ブランドコラム | 12回 | 3パターン × 4週で1巡。**ここで「また同じ構成」と感じるリスク** |
| 土:週末ピックアップ | 12回 | 6パターンで2巡。切り口が変わるので耐性あり |

**3ヶ月目のリスク: 木曜コラムの構成マンネリ**
→ 対策: 第5週目に「読者アンケート」「季節イベント特集」などの不定期ネタを挿入

### 6-2. 6ヶ月目（26週）の時点

| 曜日 | 配信済み回数 | 受信者が感じる印象 |
|------|-----------|------------------|
| 火:売れた速報 | 26回 | データが変わるので飽きにくいが「形式」には慣れる |
| 木:ブランドコラム | 26回 | 主要ブランド20〜30を1巡。**2巡目に入るブランドが出始める** |
| 土:週末ピックアップ | 26回 | 季節が一巡していないので問題なし |

**6ヶ月目のリスク: ブランドコラムの2巡目**
→ 対策: 同じブランドでも「春夏 vs 秋冬」「アウター vs トップス」で切り口を変更。AI生成プロンプトに「前回の配信内容」を含めて重複回避

### 6-3. 12ヶ月目（52週）の時点

| 曜日 | 配信済み回数 | 受信者が感じる印象 |
|------|-----------|------------------|
| 火:売れた速報 | 52回 | **形式に完全に慣れるが、数字（売約数・残数）が変動するため情報価値は維持** |
| 木:ブランドコラム | 52回 | ブランド2〜3巡目。**前年同月のデータと比較する新切り口が使えるようになる** |
| 土:週末ピックアップ | 52回 | 季節1巡。**2年目は「昨年同時期の人気アイテム」比較が可能に** |

**12ヶ月目のリスク: 全体的な形式マンネリ**
→ 対策: 1年経過時点でフォーマットをリニューアル（デザイン変更、新セクション追加）

## 8. 同じようなメールが続かないための仕組み

### 8-1. 重複防止ルール（実装必須）

```
新規メルマガ送信前チェック:

1. 今日この顧客に新着通知を送ったか？       → YES: スキップ
2. 今日この顧客に休眠クーポンを送ったか？     → YES: スキップ
3. 今日この顧客にフォローアップを送ったか？    → YES: スキップ
4. 今日は毎月1日か？                       → YES: スキップ（月次サマリー優先）
5. 今日この顧客に入金リマインドを送ったか？    → YES: スキップ
6. 前回と同じブランド/カテゴリが主題か？       → YES: 別の切り口に差し替え
```

### 8-2. コンテンツ被り防止ロジック

| 防止対象 | 方法 |
|---------|------|
| ブランドコラムの同一ブランド連続 | 過去12週の配信ブランドをScript Propertiesに保存。重複回避 |
| ピックアップの同一切り口連続 | 6パターンを順番にローテーション（インデックスを保存） |
| 売れた速報と新着通知の同日配信 | CacheServiceで当日の送信済みフラグを確認 |
| 月次サマリーとブランドコラムの内容重複 | 月初1日は新規メルマガをスキップ |
| 季節テンプレートとの重複 | 季節メルマガ（手動配信）がある週は木曜コラムを「マーケット動向」に切替 |

### 8-3. 件名の差別化ルール

| メール種別 | 件名フォーマット | 識別キーワード |
|-----------|----------------|-------------|
| 新着商品通知 | 「新着商品 ○点が入荷しました」 | **入荷** |
| 売れた速報 | 「今週○点が売約｜残り△点」 | **売約** |
| ブランドコラム | 「○○ブランドの魅力」「今月のトレンド分析」 | **ブランド名 or トレンド** |
| 週末ピックアップ | 「週末セレクト：○○特集」 | **週末セレクト** |
| 月次サマリー | 「今月の在庫サマリー」 | **在庫サマリー** |
| 休眠クーポン | 「10%OFFクーポンをお届け」 | **クーポン** |
| フォローアップ | 「ご利用ありがとうございました」 | **ありがとう** |

## 9. 最悪シナリオの検証

### 9-1. 商品在庫が極端に少ない時期

**シナリオ:** 在庫10点以下の期間が2週間続く

| メール | 影響 | 対応 |
|--------|------|------|
| 売れた速報 | 売約がほぼない → スキップ | 「在庫補充予定のお知らせ」に差し替え |
| ブランドコラム | 在庫数に依存しないため影響なし | 通常通り |
| 週末ピックアップ | 紹介できる商品が少ない | 「少数精鋭！厳選○点」として逆に希少性を訴求 |

### 9-2. 売れない時期が1ヶ月続く

| メール | 影響 | 対応 |
|--------|------|------|
| 売れた速報 | 毎週スキップされる | 週2配信（木・土）に自動縮小 |
| ブランドコラム | 「売れ筋ランキング」のデータが薄い | 「長期在庫のお買い得品」に切替 |
| 週末ピックアップ | 在庫は豊富なので問題なし | 通常通り |

### 9-3. OpenAI API障害でAI生成が使えない

| メール | 影響 | 対応 |
|--------|------|------|
| ブランドコラム | AI生成部分が使えない | テンプレート型（データのみ）にフォールバック |
| 他 | AI非依存 | 影響なし |

## 10. 年間カレンダー（月別テーマ案）

| 月 | 火:売れた速報の補足テーマ | 木:コラムの月間テーマ | 土:ピックアップの切り口 |
|----|----------------------|-------------------|---------------------|

## 11. 結論

### 1年以上持続できるか？ → **YES（条件付き）**

**持続できる理由:**
1. **売れた速報**は売約データに連動するため、ビジネスが続く限りネタは無限
2. **ブランドコラム**はブランド50+× 切り口4 = 200パターンで3.8年分の余裕
3. **週末ピックアップ**は切り口6 × 変数78 = 468パターンで9年分の余裕
4. 在庫の入れ替わりにより、同じ切り口でも中身は毎回異なる
5. 季節変動で需要品目が自然に切り替わる
6. 2年目以降は「前年比較」という新しい切り口が追加される

**必須条件:**
1. **重複防止ロジックの実装** — 既存メールとの同日配信を制御
2. **配信履歴の記録** — 過去のブランド/切り口を記録して連続回避
3. **閑散期の自動縮小** — 売約なしの週は配信をスキップまたは差し替え
4. **6ヶ月時点でのフォーマット微調整** — 構成のマンネリを防止
5. **AI生成のフォールバック** — API障害時のテンプレート型配信

**推奨アクション:**
1. まず火曜「売れた速報」から実装開始（最も実装コストが低い）
2. 2週間運用後に木曜「ブランドコラム」を追加
3. さらに2週間後に土曜「週末ピックアップ」を追加
4. 3ヶ月目に配信データを分析し、開封率の低いパターンを差し替え
