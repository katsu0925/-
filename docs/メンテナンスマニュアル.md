## clasp push / deploy の手順

### 基本フロー

```bash
# 1. ローカルからGASへプッシュ（rootDir: saisun-list を対象）
clasp push

# 2. 既存の本番デプロイを更新（必ず -i でIDを指定）
DEPLOY_ID="AKfycbzWcsi_QteRBwc2U88urRQvWG1FsrKUoFSd_r3uPmPasJnm0jfKe02IbmzlkK7Sb1x_Jg"
clasp deploy -i "$DEPLOY_ID" --description "変更内容の説明"

# 3. デプロイ一覧確認
clasp deployments
```

### clasp 設定

ルートの `.clasp.json`:
```json
{
  "scriptId": "1JKpAPuy1yEDodHe1UOg2nfPOp8HGMQuwWKxuMII4IebsSPCFuUjJRfdF",
  "rootDir": "saisun-list"
}
```

### 注意事項

- **`clasp push` だけでは本番Webアプリに反映されない**。必ず `clasp deploy -i "$DEPLOY_ID"` も実行すること
- **`clasp deploy` を `-i` なしで実行すると毎回別URLの新デプロイが作成される**。必ず上記の DEPLOY_ID を指定する
- 旧バージョンの確認は GitHub の履歴で行う

### サブプロジェクトのデプロイ

| プロジェクト | clasp設定 | scriptId |
|---|---|---|
| saisun-list (メイン) | ルート `.clasp.json` | `1JKpAPuy1y...` |
| saisun-list-bulk | `saisun-list-bulk/.clasp.json` | `15jfxP03aV...` |
| shiire-kanri | `shiire-kanri/.clasp.json` | `1q2xiZkaP0...` |

サブプロジェクトをデプロイする場合は、対象ディレクトリに移動してから `clasp push` / `clasp deploy` を実行する。

## スクリプトプロパティの設定方法

### 設定方法

**方法1: GASエディタから**
1. GASエディタを開く
2. 左メニューの歯車アイコン（プロジェクトの設定）をクリック
3. 「スクリプトプロパティ」セクションで追加・編集

**方法2: スクリプトから**
```javascript
// 個別設定
PropertiesService.getScriptProperties().setProperty('KEY', 'value');

// 一覧確認（値は先頭20文字のみ表示）
listAllProperties()  // SetupProperties.gs:96
```

### 一時プロパティのクリーンアップ

決済セッション（`PAYMENT_*`）やペンディング注文（`PENDING_ORDER_*`）は一時プロパティとして保存される。不要なものは以下で削除可能:

```javascript
// ドライラン（確認のみ、削除しない）
cleanupDryRun()          // SetupProperties.gs:215

// 実行（実際に削除）
cleanupExecute()         // SetupProperties.gs:223
```

> 毎日4時の `cronDaily4To6` で `cleanupExecute` が自動実行される。

## トリガーの設定・確認・再設定

### トリガー一覧の確認

GASエディタ → 左メニュー「トリガー」（時計アイコン）

### トリガーの全再設定

```javascript
// GASエディタから実行
setupTriggers()  // Triggers.gs:49
```

この関数は:
1. 既存トリガーをすべて削除
2. 以下を再構築:
   - onEdit トリガー × 5（注文SS、データSS、仕入れ管理SS）
   - 時間ベーストリガー × 11

### 登録されるトリガー一覧

**onEditトリガー:**
| 関数 | 対象SS | 説明 |
|---|---|---|
| `onEdit` | 注文SS | メインの編集イベントハンドラ |
| `onEdit` | データSS | 商品データ変更検知 |
| `shipMailOnEdit` | 注文SS | S列変更で発送通知 |
| `statusSync_onEdit` | 注文SS | ステータス同期 |
| `syncListingPublic` | 仕入れ管理SS | 仕入れ→データ1同期 (onEdit + onFormSubmit) |

**時間ベーストリガー:**
| 関数 | 間隔 | 説明 |
|---|---|---|
| `syncListingPublicCron` | 1分 | 仕入れ→データ1同期 |
| `cronEvery5min` | 5分 | 6関数ディスパッチャー |
| `cronAbandonedCart` | 30分 | カゴ落ち処理 |
| `cronStatsCache` | 1時間 | 統計キャッシュ |
| `cronDaily4To6` | 毎日4時 | ポイント・クリーンアップ |
| `cronDaily7` | 毎日7時 | 領収書・BASE |
| `ga4SyncAll` | 毎日8時 | GA4同期 |
| `cronDaily9` | 毎日9時 | 催促・サマリー |
| `cronNewArrival` | 毎日10時 | 新着通知 |
| `cronFollowupEmail` | 毎日11時 | フォローアップ |

### 不要トリガーの削除

スプレッドシートメニュー → 「不要トリガー一括削除」

または GASエディタで `cleanupObsoleteTriggers()` を実行。

### GASトリガーの上限

GASプロジェクトごとに最大20個のトリガーを設定可能。上限に達した場合は不要トリガーを削除してから再設定する。

## キャッシュクリア方法

### 商品データキャッシュ

```javascript
// GASエディタから実行
clearProductCache_()     // コード.gs:439
```

通常は5分ごとの `cronExportProducts` で自動更新されるが、即座に反映したい場合に使用。

### バルク商品キャッシュ

```javascript
bulk_clearCache_()       // BulkProduct.gs:104
```

### クーポンキャッシュ

クーポン登録・削除時に自動クリアされる。手動クリアが必要な場合:

```javascript
CacheService.getScriptCache().remove('COUPON_DATA_ALL');
```

### 統計キャッシュ

```javascript
// 再計算して更新
st_calculateAndCacheStats_()  // StatsCache.gs
```

### 全キャッシュ

CacheService には「全削除」メソッドがないため、個別のキーを `.remove()` で削除する必要がある。ラージキャッシュ（32MBを超えるもの）は `app_cacheDeleteLarge_()` で削除する。

## BASE OAuth トークン更新

### 自動更新フロー

1. `baseGetAccessToken_()` がAPI呼び出し時に有効期限を確認
2. 期限切れの場合、リフレッシュトークンで自動更新
3. リフレッシュトークンも期限切れの場合は手動再認証が必要

### 手動再認証

```javascript
// GASエディタから実行
baseShowAuthUrl()  // BASEAPI.gs:84
```

1. ダイアログにBASE認証URLが表示される
2. URLをコピーしてブラウザで開く
3. BASEにログインして認証を許可
4. リダイレクトでGASにコールバック → 新しいトークンが自動保存

### トークン期限アラート

毎日7時の `cronBaseTokenCheck` で残り有効期限を確認:
- 24時間以内 → 管理者にメール通知
- 期限切れ → 要対応メール通知

### 初期設定

BASE連携を初めて設定する場合:

```javascript
// GASエディタから実行
baseInitSetup()  // BASEAPI.gs:31
```

ダイアログで以下を入力:
- BASE_CLIENT_ID
- BASE_CLIENT_SECRET
- BASE_REDIRECT_URI（= GASデプロイURL /exec）
- BASE_TARGET_SS_ID

## 障害時の復旧手順

### トリガーが止まった

**症状:** 自動処理（同期・通知など）が動いていない

**復旧:**
1. GASエディタ → トリガー一覧で状態を確認
2. エラーが出ているトリガーがあれば、エラー内容を確認
3. トリガーを再設定:
```javascript
setupTriggers()  // Triggers.gs:49
```

**よくある原因:**
- GAS実行時間上限（6分）でタイムアウト
- スプレッドシートIDの変更
- 権限の変更

### Webhook が飛ばない（KOMOJU）

**症状:** KOMOJUで決済が完了しているが、依頼管理シートに注文が作成されない

**確認:**
1. KOMOJU管理画面で Webhook 送信履歴を確認
2. GASの実行ログでエラーを確認
3. `checkPendingOrders` のログを確認（5分ごとのセーフティネット）

**復旧:**
1. `KOMOJU_WEBHOOK_SECRET` を確認
```javascript
setKomojuWebhookSecret()
```
2. `DEPLOY_URL` が正しいか確認
```javascript
setDeployUrl()
```
3. KOMOJU管理画面でWebhook URLを確認:
   - 形式: `[DEPLOY_URL]?action=komoju_webhook&webhook_token=[SECRET]`

### 商品が表示されない

**症状:** サイトに商品が表示されない、または古いデータが表示される

**復旧:**
1. データ1シートの内容を確認
2. 商品キャッシュをクリア:
```javascript
clearProductCache_()
```
3. 仕入れ管理→データ1の同期を確認:
```javascript
syncListingPublicCron()
```

### メール送信上限に達した

**症状:** 「Service invoked too many times: email」エラー

**説明:** GASの1日あたりメール送信上限（Google Workspace: 1,500通、無料: 100通）

**対処:**
- 翌日に自動復旧する
- 緊急の場合は Gmail で手動送信

### スプレッドシートのデータ不整合

**症状:** 列がずれている、データが壊れている

**対処:**
1. まずスプレッドシートのバージョン履歴を確認（ファイル → 版の履歴）
2. 該当部分を復元
3. ヘッダー行が壊れた場合は `setupHeaderProtection()` を再実行

### スクリプトプロパティの容量超過

**症状:** 「ScriptProperties maximum size」エラー

**対処:**
```javascript
// まず確認
cleanupDryRun()     // 削除対象を確認

// 実行
cleanupExecute()    // 不要プロパティを削除
```

## テスト実行方法

### テストの実行手順

```javascript
// 1. 開発環境に切り替え（本番データに影響しない）
setEnvDevelopment()    // Tests.gs:62

// 2. 全10テストスイートを実行
runAllTests()          // Tests.gs:11

// 3. 本番環境に戻す（必ず忘れずに！）
setEnvProduction()     // Tests.gs:70
```

### テストスイート一覧

| スイート | 内容 |
|---|---|
| `testSuite_Auth_` | 認証・ログインテスト |
| `testSuite_Payment_` | 決済処理テスト |
| `testSuite_Order_` | 注文管理テスト |
| `testSuite_Util_` | ユーティリティ関数テスト |
| `testSuite_Integration_Auth_` | 認証統合テスト |
| `testSuite_Integration_Payment_` | 決済統合テスト |
| `testSuite_Security_` | セキュリティテスト（CSRF、ハッシュ等） |
| `testSuite_EdgeCases_` | エッジケーステスト |
| `testSuite_Rank_` | ランクシステムテスト |
| `testSuite_CsrfEnv_` | CSRF環境テスト |

### 注意事項

- **テスト実行後は必ず `setEnvProduction()` を実行すること**。忘れると本番サイトが開発モードになる
- テストはGAS上で動作するカスタム実装（外部テストフレームワークは使用していない）
- 個別スイートのみ実行する場合は、GASエディタで該当関数を直接実行

## Git 運用ルール

### ブランチ戦略

- **mainブランチで直接作業する**（フィーチャーブランチは作成しない）
- 作業完了後は自動でコミット＆プッシュする

### コミットメッセージ

変更内容を簡潔に日本語で記述する。

例:
```
欠品処理: 返品商品のステータス更新ロジックを修正
BASE同期: 画像再同期のスナップショットリセット対応
```

### リポジトリ構成

```
saisun-repo/
├── .clasp.json          ← メインプロジェクトのclasp設定
├── CLAUDE.md            ← 開発ガイドライン
├── docs/                ← ドキュメント
├── saisun-list/         ← メインプロジェクト（37+ .gsファイル）
├── saisun-list-bulk/    ← バルク商品（6 .gsファイル）
├── shiire-kanri/        ← 仕入れ管理（16 .gsファイル）
├── nkonline-official/   ← 公式サイト
├── workers/             ← Cloudflare Workers
└── tools/               ← ツール
```

### デプロイ前チェックリスト

1. [ ] テスト実行 (`setEnvDevelopment` → `runAllTests` → `setEnvProduction`)
2. [ ] `clasp push` でGASにプッシュ
3. [ ] `clasp deploy -i "$DEPLOY_ID"` で本番デプロイ
4. [ ] `clasp deployments` でデプロイ確認
5. [ ] サイトで動作確認
6. [ ] Git コミット＆プッシュ
