## 日次チェックリスト

### 朝（9:00頃）

- [ ] LINE/メールの「朝の業務サマリー」を確認（入金待ち・発送待ち・本日新規の件数）
- [ ] 依頼管理シートを開き、入金待ち注文（Q列が空）を確認
- [ ] 発送待ち注文（Q列=入金済み、S列が「発送済み」以外）を確認

### 日中

- [ ] 新規注文が入ったら依頼展開を実行（自動展開が5分ごとに動作するが、急ぎの場合は手動で）
- [ ] 欠品があれば欠品処理を実行
- [ ] 発送完了したらS列を「発送済み」に変更（→ 発送通知メールが自動送信される）

### 夕方

- [ ] 未処理の注文がないか依頼管理シートを最終確認
- [ ] コンビニ払い・銀行振込の入金期限（3日）が近い注文を確認

## 管理メニュー操作ガイド

スプレッドシートの「管理メニュー」から各機能を実行できる。

```
管理メニュー
├─ 1. 依頼展開（展開→XLSX→売却 一括処理）
├─ 2. 欠品処理（返品→再生成 一括処理）
├─ クーポン登録
├─ クーポン削除
├─ 会員割引 ON/OFF 切替
├─ 会員割引 期限変更
├─ 顧客ポイント付与（完了済み注文）
├─ 領収書送付（完了済み・希望者）
├─ 領収書取消（キャンセル/返品）
├─ 不要トリガー一括削除
└─ 運用マニュアルを開く
```

### 1. 依頼展開

**関数:** `expandOrder()` — 受注管理.gs:25

**概要:** 注文の確認ファイル生成・XLSX作成・仕入れ管理への売却反映を一括実行する。

**手順:**
1. メニューから「1. 依頼展開」を選択
2. 受付番号を入力（ダイアログが表示される）
3. 以下が自動実行される:
   - J列（選択リスト）の管理番号から確認ファイルを生成
   - 選択商品リストをExcelファイル（XLSX）にしてGoogle Driveに保存
   - 仕入れ管理のBO列に受付番号を記入（売却済みフラグ）
   - I列に確認ファイルのURLを設定

**自動展開:** 5分ごとに `cronAutoExpandOrders` が動作し、以下の条件を満たす注文を自動展開する:
- I列（確認リンク）が空
- J列（選択リスト）に値がある
- V列（ステータス）がオープン状態

### 2. 欠品処理

**関数:** `handleMissingProducts()` — 受注管理.gs:89

**概要:** 欠品商品を除外し、残りの商品で確認ファイルを再生成する。

**手順:**
1. メニューから「2. 欠品処理」を選択
2. 欠品商品の管理番号を入力（カンマ区切りで複数可）
3. 以下が自動実行される:
   - 欠品商品: 仕入れ管理でステータス→廃棄済み、廃棄日→本日、BO列クリア
   - 同一受付番号の残り商品: 売却ステータスをクリア
   - 依頼管理: 選択リスト更新（欠品を除外）、確認リンク削除
   - 自動再展開: 残りの商品で展開→XLSX→売却を再実行

### クーポン登録 / 削除

**関数:** `registerCoupon()` / `deleteCoupon()` — Coupon.gs

**クーポン管理シートの列構成:**

| 列 | 内容 | 例 |
|---|---|---|
| A | クーポンコード | SUMMER2026 |
| B | 割引タイプ | rate / fixed / shipping_free |
| C | 割引値 | rate: 0.10(=10%) / fixed: 500(=500円) / shipping_free: 0 |
| D | 有効期限 | 2026-12-31 |
| E | 利用上限 | 0(=無制限) |
| F | 利用回数 | 自動カウント |
| G | 1人1回制限 | TRUE / FALSE |
| H | 有効フラグ | TRUE / FALSE |
| I | メモ | 自由記述 |
| J | 対象顧客 | all / new / repeat |
| K | 有効開始日 | 2026-01-01 |
| L | 会員割引との併用 | TRUE / FALSE |
| M | 30点割引との併用 | TRUE / FALSE |
| N | 適用チャネル | all / detauri / bulk |
| O | 対象商品ID（bulk専用） | カンマ区切り |
| P | 送料除外商品ID | カンマ区切り |
| Q | 限定顧客名 | 特定顧客のみに適用 |
| R | 限定顧客メール | 特定顧客のみに適用 |

### 会員割引 ON/OFF 切替 / 期限変更

**関数:** `toggleMemberDiscount()` / `setMemberDiscountEndDate()` — Config.gs:213, 234

- `MEMBER_DISCOUNT_ENABLED` プロパティをトグル
- 割引率: 10%（`MEMBER_DISCOUNT_RATE` = 0.10）
- 期限設定: YYYY-MM-DD 形式で入力
- 有効期限を過ぎると自動的にOFFになる

### 顧客ポイント付与

**関数:** `processCustomerPoints()` — CustomerAuth.gs:982

**対象条件:**
- V列（ステータス）= 「完了」
- R列（ポイント付与済）が空
- D列（メール）がある
- L列（合計金額）> 0

**処理:** ランクに応じたポイント付与率 × 合計金額 でポイントを付与し、R列に「PT」を記入。

> 毎日4時にも自動実行される（`cronDaily4To6`）

### 領収書送付 / 取消

**関数:** `processInvoiceReceipts()` / `processCancelledInvoices()` — CustomerAuth.gs:1113, 1165

**送付条件:**
- Z列（インボイス発行）= 「希望」
- AA列（インボイス状況）が空
- V列（ステータス）= 「完了」

**取消条件:**
- V列 = 「キャンセル」または「返品」
- AA列 = 「送付済」

> 毎日7時にも自動実行される（`cronDaily7`）

### 不要トリガー一括削除

**関数:** `cleanupObsoleteTriggers()`

GAS の 20トリガー上限に達した場合に使用。重複した古いトリガーを削除する。

## 決済フロー（KOMOJU）

### 対応決済方法

`credit_card`, `konbini`, `bank_transfer`, `paypay`, `pay_easy`, `apple_pay`, `paidy`

定義: Constants.gs:42 (`PAYMENT_CONSTANTS.PAYMENT_METHODS`)

### 状態遷移

```
顧客が決済ページで支払い操作
│
├─ クレジットカード / PayPay / Apple Pay:
│    payment.captured → handlePaymentSuccess_() → 即座に注文確定
│
├─ コンビニ払い / 銀行振込 / ペイジー:
│    payment.authorized → handlePaymentSuccess_() → 注文作成（入金待ち）
│    │
│    ├─ 入金 → payment.updated → handlePaymentUpdated_() → Q列「入金確認」更新
│    └─ 期限切れ(3日) → payment.expired → handlePaymentFailed_() → 自動キャンセル
│
├─ 決済失敗:
│    payment.failed → handlePaymentFailed_()
│
└─ 返金:
     payment.refunded → handlePaymentRefunded_()
```

### Webhook処理

**エントリポイント:** `handleKomojuWebhook(e)` — KOMOJU.gs:207

**署名検証（2方式）:**
1. URLトークン方式（推奨）: `?action=komoju_webhook&webhook_token=SECRET`
2. HMAC-SHA256方式: `X-Komoju-Signature` ヘッダーを検証

**セーフティネット:** `checkPendingOrders()` が5分ごとに実行され、Webhookが届かなかった決済を KOMOJU API で直接確認する。3日超過のペンディング注文は自動キャンセル。

**不正対策:**
- `fetchPaymentFromApi_()` で KOMOJU API から決済情報を裏取り（Webhook改ざん検知）
- `verifyPaymentAmount_()` で金額照合（不一致時は処理中止）

### 決済期限

259,200秒（3日間） — Constants.gs:41 (`PAYMENT_CONSTANTS.PAYMENT_EXPIRY_SECONDS`)

## ランク・ポイント制度

### ランク制度

定義: CustomerAuth.gs:853-857 (`RANK_TIERS`)

判定基準: **過去12ヶ月の購入金額**（累計ではない）

| ランク | 年間購入金額 | ポイント付与率 | 送料無料 |
|---|---|---|---|
| レギュラー | 5万円未満 | 1% | - |
| シルバー | 5万円〜20万円未満 | 3% | - |
| ゴールド | 20万円〜50万円未満 | 5% | - |
| ダイヤモンド | 50万円以上 | 5% | 送料無料 |

### ポイント付与

- **タイミング:** 注文ステータスが「完了」になった後（手動 or 毎日4時自動）
- **計算:** 合計金額 × ランクに応じた付与率
- **記録:** R列に「PT」を記入（二重付与防止）

### ポイント利用

- 注文時にポイント数を入力して利用可能（1ポイント = 1円）
- 合計金額から差し引かれる

### ポイント失効

- **失効期限:** 最終ポイント更新日から2年（730日） — Constants.gs:74
- **チェック:** 毎日4時に `pointExpiryCron_()` が実行
- **リマインド:** 失効30日前に通知

## BASE連携

### 仕組み

```
デタウリ (GAS) ←→ BASE API (OAuth2.0)
  ├─ 商品同期: デタウリ → BASE（5分ごと・baseSyncProductsToBase）
  └─ 注文同期: BASE → デタウリ（5分ごと・baseSyncOrdersNow）
```

### OAuth トークン

| プロパティ | 用途 |
|---|---|
| `BASE_CLIENT_ID` | OAuthクライアントID |
| `BASE_CLIENT_SECRET` | OAuthクライアントシークレット |
| `BASE_ACCESS_TOKEN` | APIアクセストークン |
| `BASE_REFRESH_TOKEN` | リフレッシュトークン |
| `BASE_EXPIRES_AT` | トークン有効期限（タイムスタンプ） |

### トークン自動更新

- `baseGetAccessToken_()` がAPI呼び出し時に有効期限を確認
- 期限切れの場合はリフレッシュトークンで自動更新
- リフレッシュトークンも期限切れの場合は管理者にメール通知（毎日7時にチェック）
- 再認証: GASエディタで `baseShowAuthUrl()` を実行

## クーポンシステム

### 割引タイプ

| タイプ | 説明 | C列の値の例 |
|---|---|---|
| `rate` | 割合割引 | 0.10 (= 10%OFF) |
| `fixed` | 定額割引 | 500 (= 500円引き) |
| `shipping_free` | 送料無料 | 0 |

### 検証項目（フロントエンド適用時）

1. 有効期限・開始日チェック
2. 利用上限チェック
3. 1人1回制限チェック
4. 対象顧客チェック（new=新規 / repeat=リピーター / all=全員）
5. 限定顧客チェック（Q列・R列で名前+メール一致）
6. チャネル・商品チェック

### キャッシュ

クーポンデータは60秒間キャッシュされる（`COUPON_CACHE_KEY` = `COUPON_DATA_ALL`）。登録・削除時にキャッシュは自動クリアされる。

## トラブルシューティング

### 注文が届かない

| 確認ポイント | 対処法 |
|---|---|
| KOMOJU Webhookが届いていない | `checkPendingOrders` のログを確認。KOMOJU管理画面で支払い状態を確認 |
| `KOMOJU_WEBHOOK_SECRET` 未設定 | `setKomojuWebhookSecret()` をGASエディタで実行 |
| `DEPLOY_URL` が古い | スクリプトプロパティで最新のデプロイURLを設定 |

### 発送通知メールが送られない

| 確認ポイント | 対処法 |
|---|---|
| AC列に日時が入っている | 既に送信済み。再送はAC列をクリアしてS列を再設定 |
| P列（決済ID）が空 | KOMOJU経由でない注文（手動注文等）には発送通知が送られない |
| S列が正確に「発送済み」でない | 全角/半角・スペースに注意。正確に「発送済み」と入力 |

### 商品が表示されない

| 確認ポイント | 対処法 |
|---|---|
| 仕入れ管理でデータ変更後5分以上経過したか | `cronExportProducts` は5分間隔。待つか手動実行 |
| キャッシュが古い | GASエディタで `clearProductCache_()` を実行 |
| データ1シートに反映されていない | `syncListingPublicCron` のログを確認 |

### BASE連携エラー

| エラー | 対処法 |
|---|---|
| トークン期限切れ | GASエディタで `baseShowAuthUrl()` → 表示URLでBASE再認証 |
| 商品が同期されない | `baseSyncProductsToBase` のログを確認 |
| 注文が取り込まれない | `baseSyncOrdersNow` のログを確認。BASE_TARGET_SS_ID が正しいか確認 |

### スプレッドシートエラー

| エラー | 対処法 |
|---|---|
| ロック取得失敗 | 別プロセスが実行中。5〜10秒待って再実行 |
| シートが見つかりません | Config.gs のシート名設定を確認 |
| スプレッドシートが見つかりません | スクリプトプロパティのSSID設定を確認 |

### 認証エラー

| エラー | 対処法 |
|---|---|
| 権限がありません | `ADMIN_KEY` or `ADMIN_ACCESS_KEY` が不正。スクリプトプロパティを確認 |
| bot判定されました | `RECAPTCHA_SECRET` を確認。reCAPTCHAスコア閾値は0.3 |
| 管理キーが未設定です | GASエディタで `ad_initAdminOwnerAndKeyOnce()` を実行 |

### トリガーが動いていない

| 確認ポイント | 対処法 |
|---|---|
| GASエディタ → トリガー一覧で確認 | トリガーが存在しない場合は `setupTriggers()` を実行 |
| エラー通知が来ている | LINE/メールのエラー通知を確認 |
| トリガー数が20件上限に達している | 「不要トリガー一括削除」を実行してから `setupTriggers()` |
