# NotebookLM用プロンプト：在庫移動・返送管理の流れ

以下の資料をもとに、古着屋の在庫移動と返送管理の仕組みを**小学生でも分かるように**ノートブックを作成してください。ページ数は多くなっても構いません。図や表を使ってわかりやすく説明してください。

---

## 1. この仕組みは何をするものか

古着屋さんが2つのメルカリアカウントで洋服を売っています。

- **古着屋本舗**
- **ほしいが見つかる古着屋さん**

洋服は色々な場所（作業者の家など）に預けてあります。売れた分だけ新しい服を入れ替えるために、古い服を返してもらう仕組みが「入替リスト」です。また、服の保管場所が変わったときに記録するのが「移動報告」、返送されたときに記録するのが「返送管理」です。

---

## 2. 登場するシート（データの入れ物）

すべてのデータは1つのGoogleスプレッドシートの中にあります。

### 商品管理シート（メインのデータ）
すべての洋服の情報が入っている一番大事なシートです。

| 列の名前 | 何が入っているか | 例 |
|---|---|---|
| 管理番号 | 洋服ごとのID番号 | ABC-001 |
| ステータス | 今どういう状態か | 出品中、売却済み、返品済み |
| 納品場所 | 今どこにあるか（誰の家か） | 田中、佐藤 |
| 出品日 | メルカリに出した日 | 2025/1/15 |
| 販売日 | 売れた日 | 2025/2/10 |
| 使用アカウント | どっちのアカウントで出品中か | 古着屋本舗 |
| 在庫日数 | 出品してから何日たったか | 45 |

### 返送管理シート（返送の報告を書くシート）
洋服を返してもらったとき、ここに書きます。

| 列 | 何を書くか | 例 |
|---|---|---|
| A列: 箱ID | 送った箱の番号 | BOX-001 |
| B列: 報告者 | 報告した人の名前 | 田中 |
| C列: 移動先 | 返送した先（新しい場所） | 倉庫 |
| D列: 管理番号 | 返した洋服の番号（何個でもOK） | ABC-001, ABC-002 |
| E列: 着数 | 何着返したか | 5 |
| F列: 備考 | メモ | 汚れあり1点 |

### 移動報告シート（場所が変わったときに書くシート）
洋服の保管場所が変わったとき、ここに書きます。AppSheet（スマホアプリ）から入力します。

| 列 | 何を書くか | 例 |
|---|---|---|
| A列: 移動ID | 自動でつく番号 | MV-20250210-093015 |
| B列: タイムスタンプ | いつ報告したか | 2025/2/10 9:30 |
| C列: 報告者 | 報告した人 | 佐藤 |
| D列: 移動先 | 新しい場所 | 鈴木 |
| E列: 管理番号 | 移動した洋服の番号（何個でもOK） | ABC-003, ABC-004 |
| F列: 処理済み | コンピュータが処理したか | TRUE |

### 作業者マスター（作業者の名簿）
洋服を預かっている人の情報です。

| 列 | 何を書くか |
|---|---|
| 名前 | 作業者の名前 |
| 入替除外 | TRUEなら入替リストの対象外にする |

### 在庫分析シート
回収状況の分析データです。回収割合が一定以上になったら「回収完了日」が自動で入ります。

## 3. 洋服のステータス（状態）の種類

洋服は今どういう状態かを「ステータス」で管理しています。

| ステータス | 意味 |
|---|---|
| **出品中** | メルカリに出している状態。お客さんが買えるよ |
| **売却済み** | 売れた！お客さんが買ってくれた |
| **発送待ち** | 売れて、これから送るところ |
| **発送済み** | お客さんに送った |
| **返品済み** | 作業者から返してもらった |
| **キャンセル済み** | 注文がキャンセルになった |
| **廃棄済み** | もう捨てた |

---

## 4. ステータスがどう変わるか（状態の変化）

```
出品中（メルカリで売ってる）
  │
  ├── 売れた！ → 売却済み → 発送待ち → 発送済み
  │
  ├── 入替リストに載った → 返してもらう → 返送管理に記入
  │     → 自動で「返品済み」に変わる
  │
  ├── もう売れない → 廃棄済み
  │
  └── キャンセルされた → キャンセル済み
```

**大事なポイント:** 「返品済み」に自動で変わるのは、返送管理シートに管理番号を書いたときです。人がステータスを手で変える必要はありません。

## 5. 入替リストの仕組み（月末の入れ替え作業）

### いつ動くの？
毎月28日の朝9時に自動で動きます。

### 何をするの？
「先月売れた数」と同じ数だけ、今出品中の古い洋服を選んで「返してね」リストを作ります。

### 具体例で説明

**古着屋本舗の場合：**

1. まず「先月何着売れたか」を数えます
   - 先月（1月）にステータスが「売却済み」で、販売日が1月1日〜1月31日の洋服を数えます
   - → **18着売れた！**

2. 次に「出品中」の洋服を、出品日が古い順に並べます
   - 一番古い: ABC-001（出品日: 2024/6/1）
   - 2番目: ABC-015（出品日: 2024/6/15）
   - 3番目: ABC-022（出品日: 2024/7/1）
   - ...

3. 古い順に **18着**（売れた数と同じ）を選びます
   - → これが「入替リスト」になります

4. PDFファイルを作って、メールで送ります

### 除外される作業者
作業者マスターで「入替除外」がTRUEになっている人の洋服は、入替リストの対象から外されます。たとえばオーナーの「かつ」さんの洋服は入替の対象にしない、などです。

### PDF（入替リスト）の中身

```
入替リスト — 古着屋本舗
集計期間: 2025年1月販売分（2025/1/1〜2025/1/31）
前月販売数: 18件
返送対象: 18件（出品日 2024/6/1 〜 2024/8/20）

No.  管理番号    納品場所  使用アカウント    出品日
1    ABC-001    田中      古着屋本舗        2024/6/1
2    ABC-015    佐藤      古着屋本舗        2024/6/15
3    ABC-022    田中      古着屋本舗        2024/7/1
...
18   ABC-099    鈴木      古着屋本舗        2024/8/20
```

### メールの中身
- **宛先:** スクリプトプロパティに設定されたメールアドレス
- **件名:** 【入替リスト】古着屋本舗 2025年1月分 — 18件
- **本文:** 前月販売数18件、返送対象18件、出品日範囲
- **添付ファイル:** 上のPDF

## 6. 返送管理の仕組み（返してもらったとき）

### いつ動くの？
- 返送管理シートに何か書いたら**すぐに**自動で動きます（onChangeトリガー）
- 念のため**1時間ごと**にもチェックします

### 何をするの？
返送管理シートに書かれた管理番号を見て、商品管理シートの情報を自動で更新します。

### 処理の流れ

```
① 作業者が返送管理シートに記入
   箱ID: BOX-005
   報告者: 田中
   移動先: 倉庫
   管理番号: ABC-001, ABC-015, ABC-022

② コンピュータが自動で検知（onChange）

③ 管理番号を分解（カンマ、改行、スペースで区切られたものを分ける）
   → ABC-001、ABC-015、ABC-022 の3つ

④ 商品管理シートでこの3つを探す

⑤ 見つかった商品のステータスを「返品済み」に変更
   ※ ただし以下のステータスの場合は変更しない：
   - 売却済み
   - 廃棄済み
   - キャンセル済み
   - 発送待ち
   - 発送済み

⑥ 移動先（この例では「倉庫」）が入力されていれば、
   商品管理シートの「納品場所」も「倉庫」に更新
```

### 管理番号の入力方法（どれでもOK）
```
カンマ区切り:  ABC-001, ABC-002, ABC-003
改行区切り:    ABC-001
               ABC-002
               ABC-003
スペース区切り: ABC-001 ABC-002 ABC-003
```

## 7. 移動報告の仕組み（保管場所が変わったとき）

### いつ動くの？
移動報告シートに何か書いたら**すぐに**自動で動きます（onChangeトリガー）。

### 何をするの？
洋服の保管場所が変わったとき（たとえば田中さんの家から鈴木さんの家に移ったとき）、商品管理シートの「納品場所」を自動で更新します。

### 処理の流れ

```
① 作業者がAppSheet（スマホアプリ）から移動報告を入力
   報告者: 佐藤
   移動先: 鈴木
   管理番号: ABC-050, ABC-051

② 移動報告シートに自動で行が追加される
   移動ID: MV-20250210-093015（自動生成）
   処理済み: （まだ空）

③ コンピュータが自動で検知（onChange）

④ 処理済みが「TRUE」でない行を探す

⑤ 管理番号を分解
   → ABC-050、ABC-051

⑥ 商品管理シートでこの2つを探す

⑦ 見つかった商品の「納品場所」を「鈴木」に更新

⑧ 処理済みに「TRUE」を書き込み（次は処理しない）
```

### 返送管理との違い
| | 移動報告 | 返送管理 |
|---|---|---|
| **何が変わる？** | 納品場所だけ | ステータス＋納品場所 |
| **ステータス** | 変わらない（出品中のまま） | 「返品済み」に変わる |
| **使う場面** | 場所が変わっただけ | 返送してもらったとき |
| **入力方法** | AppSheet（スマホ） | スプレッドシートに直接 |

## 8. 在庫日数の自動計算

### いつ動くの？
毎日朝4時に自動で動きます。

### 何をするの？
出品してから何日たったかを自動で計算して、商品管理シートの「在庫日数」列に書き込みます。

### 計算方法
```
在庫日数 = 今日の日付 − 出品日

例：今日が2025年2月25日、出品日が2025年1月10日の場合
   在庫日数 = 46日
```

### 在庫日数が0になるステータス
以下のステータスの洋服は在庫ではないので、在庫日数は0になります：
- 発送済み
- 発送待ち
- 売却済み
- キャンセル
- 返品済み
- 廃棄済み

---

## 9. 回収完了の自動スタンプ

### いつ動くの？
毎日午前0時30分に自動で動きます。

### 何をするの？
在庫分析シートで「回収割合」が一定の数字（閾値）以上になった商品に、「回収完了日」を自動で書き込みます。

### 例
閾値が80%の場合：
- 回収割合が85% → 回収完了日に今日の日付を書き込む
- 回収割合が50% → まだ書き込まない

## 10. 全体の流れをまとめた図

```
【毎月の入替サイクル】

  商品を仕入れる
      ↓
  作業者に預ける（納品場所を記録）
      ↓
  メルカリに出品する（ステータス：出品中）
      ↓
  ┌──────────────────────────────────┐
  │  出品中の期間（毎日自動で在庫日数を計算）│
  │                                    │
  │  場所が変わったら → 移動報告        │
  │  （納品場所が自動で更新される）      │
  └──────────────────────────────────┘
      ↓
  お客さんが買ってくれる（ステータス：売却済み）
      ↓
  発送する（ステータス：発送済み）
      ↓
  ┌──────────────────────────────────┐
  │  毎月28日 朝9時 入替リスト自動生成  │
  │                                    │
  │  先月売れた数を数える               │
  │       ↓                            │
  │  同じ数だけ、古い出品中の洋服を選ぶ │
  │       ↓                            │
  │  PDFを作ってメールで送る            │
  └──────────────────────────────────┘
      ↓
  作業者が入替リストの洋服を返送する
      ↓
  返送管理シートに管理番号を記入
      ↓
  ┌──────────────────────────────────┐
  │  自動処理（すぐ or 1時間以内）      │
  │                                    │
  │  ステータス → 「返品済み」に変更    │
  │  納品場所 → 移動先に変更            │
  └──────────────────────────────────┘
      ↓
  返品済みの洋服の代わりに新しい洋服を出品
      ↓
  最初に戻る（サイクル）
```

## 12. よくある質問

### Q: 入替リストに載った洋服は自動で返品済みになるの？
**A: いいえ。** 入替リストはあくまで「返してね」という依頼のリストです。実際に返送されたら、返送管理シートに管理番号を記入します。そのタイミングで自動的に「返品済み」に変わります。

### Q: 移動報告と返送管理の違いは？
**A:** 移動報告は「場所が変わっただけ」（まだ出品中）。返送管理は「返してもらった」（返品済みになる）。

### Q: 入替除外ってなに？
**A:** 作業者マスターで「入替除外＝TRUE」にすると、その人が預かっている洋服は入替リストに載りません。例えばオーナーが自分で管理している洋服など。

### Q: 管理番号を間違えて入力したらどうなる？
**A:** 商品管理シートに存在しない管理番号は無視されます（ログに記録されます）。

### Q: 前月に1着も売れなかったらどうなる？
**A:** 入替リストは0件になり、PDFもメールも送られません。

## 13. 技術的な補足（大人向け）

### 使っている技術
- **Google Apps Script (GAS):** 自動処理を動かすプログラム
- **Google Sheets:** すべてのデータを保存するデータベース代わり
- **AppSheet:** スマホから移動報告を入力するアプリ
- **Gmail:** 入替リストのPDFを送るメール

### トリガーの種類
- **onChange:** シートに何か変更があったらすぐ実行
- **TimeBased:** 決まった時間に自動実行（毎日、毎時、毎月など）

### ロック機構
同時に複数の処理が走ると問題になるので、返送管理と移動報告は「ロック」をかけて、1つずつ順番に処理します（最大25秒）。

|---|---|

| ほしいが見つかる古着屋さん | SWAP_EMAIL_HOSHIIGA |
